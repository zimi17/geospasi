# ADR-001: Keputusan Arsitektur GeoSERANG

**Tanggal:** 2026-06-07
**Status:** Disetujui
**Pemrakarsa:** Arsitek & Product Manager

---

## 1. Database: Supabase PostGIS (Cloud) — Bukan Native/Docker

**Konteks:** Mesin pengembangan RAM 4GB tidak kuat menjalankan Docker. Target kontributor termasuk mahasiswa dengan laptop terbatas. Data fase 0-1 masih kecil (<500 MB).

**Keputusan:** Gunakan Supabase (PostGIS + PostgreSQL) free tier sebagai database default. Koneksi via connection string di `.env`. Bisa ditukar ke koneksi lokal di fase 2+.

**Dampak:**
- (+) 10 menit setup, tanpa instalasi server database
- (+) Kontributor bisa akses dari mana saja tanpa konfigurasi jaringan
- (+) Free tier 500 MB memaksa selektif terhadap data yang dimuat
- (-) Ekstensi PostGIS lanjutan (routing, topology) mungkin terbatas
- (-) Data tidak bisa diakses offline

---

## 2. Struktur Folder: Domain-Driven dengan Nama Bahasa Indonesia

**Konteks:** Proyek akan memiliki banyak modul analisis (pajak restoran, PBB, hotel, hiburan). Kontributor termasuk akademisi/mahasiswa non-teknis.

**Keputusan:** Struktur berdasarkan domain, bukan teknis. Setiap modul memiliki folder `data/`, `etl/`, `analysis/`, `web/` sendiri. Nama folder dalam Bahasa Indonesia (contoh: `potensi_pajak_restoran/`).

**Dampak:**
- (+) Setiap modul mandiri — kontributor cukup masuk ke satu folder
- (+) Menambahkan modul baru tidak perlu merombak struktur
- (+) Non-teknis langsung paham navigasi tanpa jargon asing
- (-) Ada duplikasi struktur antar modul (trade-off yang diterima)

---

## 3. Versioning Data: CSV & GeoJSON Kecil di Repo — Bukan Git LFS/DVC

**Konteks:** Data spasial bisa membesar. Git LFS punya kuota terbatas, DVC punya kurva belajar. Fase 0 hanya butuh data contoh <500 KB.

**Keputusan:** File CSV dan GeoJSON <1 MB disimpan langsung di Git. File >1 MB masuk `.gitignore`, disimpan di PostGIS atau Google Drive dengan tautan di README. DVC dipertimbangkan di fase 2+.

**Dampak:**
- (+) `git clone` langsung dapat data sampel — tanpa setup tambahan
- (+) Kontributor non-teknis tidak perlu belajar LFS/DVC
- (+) Repo tetap ringan (<1 MB data sampel)
- (-) Data besar tidak ter-versioning sampai fase 2

---

## 4. ETL: Satu Script Modular — Bukan Monolitik atau Microservice

**Konteks:** Fase 0 hanya ada satu sumber data (CSV restoran). Ke depan akan ada data hotel, PBB, BPS, OSM.

**Keputusan:** Satu file (`ingest_restoran.py`) dengan fungsi-fungsi terpisah (`load_config`, `read_csv`, `validate`, `transform`, `load_to_postgis`). Fungsi umum akan diekstrak ke `shared/etl_utils.py` jika digunakan modul lain.

**Dampak:**
- (+) Satu file mudah dibaca, alur ETL jelas dari atas ke bawah
- (+) Fungsi sudah terisolasi — refactor ke shared mudah dilakukan
- (+) Mahasiswa bisa memahami alur ETL tanpa lompat file
- (-) Fungsi shared baru diekstrak ketika benar-benar dibutuhkan

---

## 5. Frontend: Static Leaflet.js — Bukan React/Vue/Framework

**Konteks:** Web publik untuk menampilkan peta hasil analisis. Target kontributor termasuk mahasiswa yang hanya tahu HTML dasar.

**Keputusan:** Gunakan HTML + CSS + JavaScript vanilla dengan Leaflet.js. Hosting di GitHub Pages. Tiga file terpisah (`index.html`, `map.js`, `style.css`). Zero build step, zero dependency management.

**Dampak:**
- (+) Bisa diedit dengan teks editor apa pun, tanpa Node.js/npm
- (+) Push ke GitHub, langsung live — tidak ada proses build
- (+) Mahasiswa cukup edit HTML untuk mengubah konten peta
- (-) Filter interaktif dan chart kompleks ditunda ke fase 2+

---

## 6. QGIS Model: Hybrid (File .model3 + Dokumentasi Langkah)

**Konteks:** QGIS Processing Model (.model3) adalah XML yang bisa langsung di-load. Tapi rentan rusak jika nama layer berubah, dan sulit di-code-review.

**Keputusan:** Simpan `.model3` di repo untuk kemudahan load + dokumentasi langkah demi langkah di `analysis/README.md` untuk transparansi metodologi.

**Dampak:**
- (+) Mahasiswa bisa langsung load dan run model tanpa membangun dari nol
- (+) Dokumentasi menjelaskan *mengapa*, bukan hanya *bagaimana*
- (+) Jika `.model3` rusak, dokumentasi jadi panduan rekonstruksi
- (-) Dua sumber kebenaran yang harus dijaga konsistensinya

---

## 7. Geocoding: Hybrid Nominatim + Validasi Manual Mahasiswa

**Konteks:** Data CSV dari Dinas hanya berisi alamat, tanpa koordinat. Nominatim gratis tapi akurasinya tidak selalu baik untuk alamat desa.

**Keputusan:** Script ETL melakukan geocoding otomatis via Nominatim, memberi confidence score ("tinggi", "sedang", "rendah"). Mahasiswa memvalidasi titik yang confidence-nya rendah di QGIS secara manual.

**Dampak:**
- (+) Biaya nol rupiah untuk geocoding 50-100 titik
- (+) Mahasiswa mendapat pengalaman validasi data spasial langsung
- (+) Confidence score menjadi bagian dari layer kredibilitas data
- (-) Proses validasi manual butuh waktu 1-2 jam per batch

---

## 8. Keamanan: .env + Supabase RLS — Bukan IP Whitelist

**Konteks:** Script ETL perlu koneksi ke PostGIS. Service key admin tidak boleh bocor ke publik. Frontend publik hanya perlu akses baca data agregat.

**Keputusan:** File `.env` menyimpan credentials (tidak pernah di-commit). Supabase RLS diaktifkan: anon key (publik) hanya bisa SELECT dari tabel publik; service role bisa INSERT/UPDATE/DELETE. Template SQL policy disediakan di `shared/sql/rls_policies.sql`.

**Dampak:**
- (+) Anon key aman dipakai di frontend Leaflet.js
- (+) Jika `.env` tidak sengaja ter-commit, hanya anon key yang bocor
- (+) Mahasiswa belajar praktik keamanan database standar industri
- (-) Setup RLS butuh pemahaman SQL policy (tapi template sudah disediakan)

---

## 9. Validasi: Bertingkat dengan Output Laporan — Bukan Gagal Total

**Konteks:** Data pemerintah sering kotor (omzet berupa range, NPWP kosong, alamat tidak terstruktur). ETL akan gagal jika hanya mengandalkan parsing sederhana.

**Keputusan:** Validasi bertingkat di setiap kolom. Data bermasalah tetap dimuat ke database dengan flag confidence (`high`/`low`/`unparseable`) dan catatan di kolom `validation_notes`. Script menghasilkan dua file: data bersih + laporan masalah CSV.

**Dampak:**
- (+) Tidak ada data yang dibuang — masalah didokumentasikan, bukan disembunyikan
- (+) Laporan masalah menjadi artefak riset yang bisa dikutip di skripsi
- (+) Mahasiswa belajar bahwa data cleaning adalah 80% pekerjaan analis
- (-) Ada data berkualitas rendah di database yang harus difilter saat analisis

---

## 10. Lisensi & Tata Kelola Data: Kode Open Source, Data Bertingkat

**Konteks:** Kode harus terbuka untuk kolaborasi. Tapi data pajak asli (nama, NPWP, omzet restoran) tunduk pada UU PDP dan perjanjian dengan Pemkab.

**Keputusan:**
- **Kode:** AGPLv3 (shared), MIT (template analisis) — publik di GitHub
- **Data sintetis:** CC0 (domain publik) — di repo sebagai sampel
- **Data agregat:** Publik via Supabase anon key — ringkasan per desa tanpa identitas individu
- **Data mentah:** Akses terbatas peneliti terdaftar — di schema `restricted`, tidak pernah masuk Git

**Dampak:**
- (+) Kode tetap terbuka untuk kolaborasi global
- (+) Data sensitif terlindungi sesuai UU PDP
- (+) Reproducibility terjamin via data sintetis yang strukturnya identik
- (-) Peneliti baru perlu melalui proses persetujuan untuk akses data mentah
